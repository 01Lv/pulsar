Understand schema

本章解釋了 Pulsar schema 的基本概念，討論了一些特別重要的專題，並提供了額外的背景知識。

SchemaInfo

Pulsar schema 使用 SchemaInfo 來定義數據結構 。

SchemaInfo 是在每個 topic 的基礎上存儲和執行的，它不能設定在命名空間或租戶層級。

SchemaInfo 由以下元件組成：

<th>
  Description
</th>

<td>
  Schema 名稱 (字串)。
</td>

<td>
  Schema 類型，它決定如何解釋 schema 數據。 * 預定義 schema: 詳情參考 [schema-type](/docs/zh-TW/2.4.1/schema-understand#schema-type)。 * 自定義 schema：空字串。
</td>

<td>
  Schema 數據是一個由 8-bit unsigned bytes 和特定 schema-type 組成的序列。
</td>

<td>
  這是一個使用者定義的屬性作為一個 string/string map。 Applications can use this bag for carrying any application specific logics. 屬性可能是與 schema 相關聯的 Git 哈希，一個環境字串，如`dev` 或 `prod` 。
</td>
Field
`name`
`type`
`schema`（`payload`)
`properties`

Example

這是 STRING 類型的 SchemaInfo。

{
    “name”: “test-string-schema”,
    “type”: “STRING”,
    “schema”: “”,
    “properties”: {}
}

Schema type

Pulsar 支持各種 schema 型別，主要分為兩類：

  • 原始型別 (Primitive type)

  • 複雜型別 (Complex type)

原始型別 (Primitive type)

目前，Pulsar 支持以下原始型別：

原始型別Description
BOOLEAN二進制值
INT8一個 8 位符號整數
INT16一個 16 位符號整數
INT32一個 32 位符號整數
INT64一個 64 位符號整數
FLOAT單精度(32位) IEEE 754 浮點數
DOUBLE雙精度(64位) IEEE 754 浮點數
BYTES8 位無符號字節的序列
STRINGUnicode 字符序列
TIMESTAMP (DATE, TIME)邏輯類型代表時間內特定的即時精度毫秒。
它儲存自 1970年1月1日，00:00:00 GMT 以來的毫秒數為 INT64 值

對於原始型別，Pulsar 不在 SchemaInfo 中存儲任何 schema 數據。 SchemaInfo 中的 type 用於決定如何序列化和反序列化數據。

一些原始型別的實現可以使用 properties 來存儲實現特定的調校設定。 例如， STRING schema 可以使用 properties 來存儲字符編碼集來序列化和反序列化字符串。

Pulsar schema 型別和特定語言原始資料型別之間的對應關係如下所示。

Schema 型別Java 型別Python TypeGo 型別
BOOLEANbooleanboolbool
INT8byteint8
INT16shortint16
INT32intint32
INT64longint64
FLOATfloatfloatfloat32
DOUBLEdoublefloatfloat64
BYTESbyte[], ByteBuffer, ByteBufbytes[]byte
STRINGstringstrstring
TIMESTAMPjava.sql.Timestamp
TIMEjava.sql.Time
DATEjava.util.Date

Example

此範例演示如何使用 STRING schema。

  1. 創建一個指定使用 STRING schema 的 producer 並發送信息。

    Producer producer = client.newProducer(Schema.STRING).create();
producer.newMessage().value("Hello Pulsar!").send();

  2. 使用 STRING schema 創建 consumer 並接收信息。

    Consumer consumer = client.newConsumer(Schema.STRING).create();
consumer.receive();

複雜型別 (Complex type)

目前，Pulsar支持以下複雜型別:

複雜型別Description
keyvalue表示一個複雜型別的鍵/值對。
struct支持 AVRO, JSON, 和 Protobuf。

keyvalue

keyvalue schema 幫助應用程序定義 schema 的鍵和值。

對於 keyvalue schema 的 schemaInfo 保存策略, Pulsar 會把 key schema 的 SchemaInfo 和value schema 的 SchemaInfo 給儲存下來。

Pulsar 提供兩種方法來編碼信息中的鍵/值對：

  • INLINE

  • SEPARATED

使用者可以在構建 keyvalue schema 時選擇編碼類型。

INLINE

鍵/值對將在信息 payload 中一起編碼存放。

SEPARATED

key 將被編碼為信息 key，其 value 將被編碼為信息的 payload。

Example

這個範例演示如何構建一個 key/value schema，然後用它來生成和消費信息。

  1. 構建具有 INLINE 編碼類型的 key/value schema。

    Schema<KeyValue<Integer, String>> kvSchema = Schema.KeyValue(
    Schema.INT32,
    Schema.STRING,
    KeyValueEncodingType.INLINE
);

  2. 或是構建一個包含 SEPARATED 編碼類型的 key/value schema。

    Schema<KeyValue<Integer, String>> kvSchema = Schema.KeyValue(
    Schema.INT32,
    Schema.STRING,
    KeyValueEncodingType.SEPARATED
);

  3. 使用 key/value 生成信息。

    Schema<KeyValue<Integer, String>> kvSchema = Schema.KeyValue(
Schema.INT32,
Schema.STRING,
KeyValueEncodingType.SEPARATED
);

Producer<KeyValue<Integer, String>> producer = client.newProducer(kvSchema)
    .topic(TOPIC)
    .create();

final int key = 100;
final String value = "value-100”;

// send the key/value message
producer.newMessage()
.value(new KeyValue<>(key, value))
.send();

  4. 使用 key/value schema 消費信息。

    Schema<KeyValue<Integer, String>> kvSchema = Schema.KeyValue(
    Schema.INT32,
    Schema.STRING,
    KeyValueEncodingType.SEPARATED
);

Consumer<KeyValue<Integer, String>> consumer = client.newConsumer(kvSchema)
    ...
    .topic(TOPIC)
    .subscriptionName(SubscriptionName)
    .subscribe();

// receive key/value pair
Message<KeyValue<Integer, String>> msg = consumer.receive();
KeyValue<Integer, String> kv = msg.getValue();

struct

Pulsar 使用 Avro 規格 來定義 struct 的 schema。

Pulsar 允許：

  • 使用相同的工具管理 schema 定義

  • 使用不同的序列化/反序列化方法處理數據

有兩種方法可以使用 struct schema：

  • static

  • generic

static

你可以預定義 struct schema，它可以是 Java 中的 POJO 類別、Go 的 struct, 或透過 Avro 或 Protobuf 工具所生成的類別。

Example

Pulsar 使用 Avro 來定義 struct 類型的 schema。 schema 定義是存放在 SchemaInfo 裡 schema 欄位。

  1. 創建 User 類別來定義要發送到 Pulsar topic的信息資料類別。

    public class User {
    String name;
    int age;
}

  2. 創建一個 struct schema 的 producer 並發送信息。

    Producer<User> producer = client.newProducer(Schema.AVRO(User.class)).create();

producer.newMessage()
        .value(User.builder()
                   .userName("pulsar-user")
                   .userId(1L)
                   .build())
        .send();

  3. 創建一個 struct schema的 consumer 並接收信息。

    Consumer<User> consumer = client.newConsumer(Schema.AVRO(User.class)).create();
User user = consumer.receive();
generic

有時應用程序沒有預定義的結構，你也可以使用此方法來定義 schema 和訪問數據。

你可以使用 GenericSchemaBuilder 來定義 struct schema。使用 GenericSchemaBuilder 生成一個generic struct，並將消息轉換成 GenericRecord。

Example

  1. 使用 RecordSchemaBuilder 來構建一個 schema。

    RecordSchemaBuilder recordSchemaBuilder = SchemaBuilder.record("schemaName");
recordSchemaBuilder.field("intField").type(SchemaType.INT32);
SchemaInfo schemaInfo = recordSchemaBuilder.build(SchemaType.AVRO);

Producer producer = client.newProducer(Schema.generic(schemaInfo)).create();

  2. 使用 RecordBuilder 來構建 struct record。

    producer.newMessage().value(schema.newRecordBuilder()
            .set("intField", 32)
            .build()).send();

Auto Schema

如果你事先不知道 Pulsar topic 裡頭資料的 schema，你可以使用 AUTO schema 來對 broker 生產或消費 generic record。

Auto Schema 類型Description
AUTO_PRODUCE這有助於將有 schema 的數據從 producer 發佈到 Pulsar topic。
AUTO_CONSUME這有助於將有 schema 的數據從 Pulsar topic 傳送到 consumer。

AUTO_PRODUCE

AUTO_PRODUCE schema 有助於 producer 來驗證要發送的信息 bytes 是否與topic預定義的 schema 相容。

Example

假設：

  • 你需要處理從 Kafka topic K 來的信息。

  • 你有一個 Pulsar topic P，而且你不知道它預定義的 schema。

  • 你的應用程序要負責讀取來自 K 的信息，然後寫入到 P。

在這種情況下， 你可以使用 AUTO_PRODUCE 來驗證由 K 生成的 bytes 是否可以發送到 P。

Produce pulsarProducer = client.newProducer(Schema.AUTO_PRODUCE())
    …
    .create();

byte[] kafkaMessageBytes = … ; 

pulsarProducer.produce(kafkaMessageBytes);

AUTO_CONSUME

AUTO_CONSUME schema 有助 Pulsar 驗證由 Pulsar topic 發送的 bytes 是否與 consumer 指定的 schema 相容，也就是說 Pulsar 會使用儲存在 topic 裡的 SchemaInfo 來將信息反序列化為特定語言的物件。

目前 AUTO_CONSUME 只支持 AVRO 和 JSON schema。 它將信息反序列化為 GenericRecord。

Example

假設：

  • 你有一個 Pulsar topic P。

  • 你有一個 consumer (要把資料轉入資料庫，例如 MySQL) 消費 topic P 的信息。

  • 你的應用程式讀取來自 P 的信息，並將信息寫入 MySQL。

在這種情況下， 你可以使用 AUTO_CONSUME 來驗證由 P 生成的 bytes 是否可以發送到 MySQL 。

Consumer pulsarConsumer = client.newConsumer(Schema.AUTO_CONSUME())
    …
    .subscribe();

Message msg = consumer.receive() ; 
GenericRecord record = msg.getValue();
…

Schema 版本

存儲在 topic 的 SchemaInfo 都有版本號。 Schema 版本管理一個 topic 的 schema 變動。

指定 SchemaInfo 所產生的信息會有schema版本的標記，這樣當信息被 Pulsar 客戶端消費時，Pulsar 客戶端可以使用 schema 版本來取回相對應的 SchemaInfo，然後使用它去反序列化數據。

Schemas are versioned in succession. Schema 存儲在 broker 中，以便幫助 topic 來進行版本的標記與指派。

Once a version is assigned/fetched to/for a schema, all subsequent messages produced by that producer are tagged with the appropriate version.

Example

下面的範例說明 schema 版本是如何作用的。

假設使用 Pulsar Java 客戶端 使用下面的程式碼連接到 Pulsar 並開始發送信息：

PulsarClient client = PulsarClient.builder()
        .serviceUrl("pulsar://localhost:6650")
        .build();

Producer producer = client.newProducer(JSONSchema.of(SensorReading.class))
        .topic("sensor-data")
        .sendTimeout(3, TimeUnit.SECONDS)
        .create();

下表列出了跟 schema 相關的使用情境：

<th>

</th>

<td>

</td>

</td>

<tr>

  </td>

<h2>
  Schema 是如何運作的
</h2>

<p>
  Pulsar schema 在 <strong>topic</strong> 層級應用和執行 (不能在定義在命名空間或租戶層級)。
</p>

<p>
  Producer 和 consumer 將 schema 上傳給 broker，因此 Pulsar schema 會在 producer 和 consumer 兩方面發揮作用。
</p>

<h3>
  Producer side
</h3>

<p>
  這個圖表說明 schema 在 producer 這一方的運作方式。
</p>

<p>
  <img src="assets/schema-producer.png" alt="Schema works at the producer side" />
</p>

<ol start="1" spaces="0" level="0">
  <li level="0">
    <p>
      應用程序使用 schema 實例構建 producer 實例。
    </p>
    <p>
      Schema 實例定義了 producer 實例生成的數據資料型別 schema。
    </p>
    <p>
      以AVRO為範例，Pulsar 從 POJO 類別中提取 schema 定義，並構建 producer 連接時需要傳遞給 broker 的 <code>SchemaInfo</code>。
    </p>
  </li>
  
  <li level="0">
    <p>
      Producer 連接到 broker 並傳送前一步驟所提取出來的 <code>SchemaInfo</code>。
    </p>
  </li>
  
  <li level="0">
    <p>
      Broker 在 schema 存儲中查找 schema 以檢查它是否已經是註冊過的 schema。
    </p>
  </li>
  
  <li level="0">
    <p>
      如果是，broker 會跳過 schema 驗證，因為它是一個已知的schema，並將 schema 版本號回覆給 producer。
    </p>
  </li>
  
  <li level="0">
    <p>
      If no, the broker validates the schema based on the schema compatibility check strategy defined for the topic.
    </p>
  </li>
  
  <li level="0">
    <p>
      If the schema is compatible, the broker stores it and returns the schema version to the producer.
    </p>
    <p>
      All the messages produced by this producer are tagged with the schema version.
    </p>
  </li>
  
  <li level="0">
    <p>
      If the schema is incompatible, the broker rejects it.
    </p>
  </li>
</ol>

<h3>
  Consumer side
</h3>

<p>
  This diagram illustrates how does Schema work on the consumer side.
</p>

<p>
  <img src="assets/schema-consumer.png" alt="Schema works at the consumer side" />
</p>

<ol start="1" spaces="0" level="0">
  <li level="0">
    <p>
      The application uses a schema instance to construct a consumer instance.
    </p>
    <p>
      The schema instance defines the schema that the consumer uses for decoding messages received from a broker.
    </p>
  </li>
  
  <li level="0">
    <p>
      The consumer connects to the broker with the <code>SchemaInfo</code> extracted from the passed-in schema instance.
    </p>
  </li>
  
  <li level="0">
    <p>
      Broker 在 schema 存儲中查找 schema 以檢查它是否已經是註冊過的 schema。
    </p>
  </li>
  
  <li level="0">
    <p>
      If yes, the broker skips the schema validation since it is a known schema, and returns the schema version to the consumer.
    </p>
  </li>
  
  <li level="0">
    <p>
      If no, the broker validates the schema based on the schema compatibility check strategy defined for the topic.
    </p>
  </li>
  
  <li level="0">
    <p>
      If the schema is compatible, the broker stores it and returns the schema version to the consumer.
    </p>
  </li>
  
  <li level="0">
    <p>
      If the schema is incompatible, the consumer will be disconnected.
    </p>
  </li>
  
  <li level="0">
    <p>
      The consumer receives the messages from the broker.
    </p>
    <p>
      If the schema used by the consumer supports schema versioning (for example, AVRO schema), the consumer fetches the <code>SchemaInfo</code> of the version tagged in messages, and use the passed-in schema and the schema tagged in messages to decode the messages.
    </p>
  </li>
</ol>
