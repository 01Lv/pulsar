Understand schema

本章解释了Pulsar schema 的基本概念，重点讨论了一些特别重要的专题，并提供了额外的背景知识。

SchemaInfo

在 SchemaInfo 的数据结构中定义了 Pulsar schema 。

SchemaInfo 是在每个主题的基础上存储和执行的，不能存储在命名空间或租户一级。

SchemaInfo 由以下字段组成：

字段

Description

name

Schema 名称 (字符串)。

type

Schema 类型，它决定如何解释 schema 数据。

  • 预定义 schema: 详情参考 here.

  • 自定义模式：留空字符串。

schema（payload)

Schema 数据是一个由 8 位无符号字节和模式类型特定组成的序列。

properties

这是一个用户定义的属性作为字符串/字符串地图。

应用可以使用这个包裹传递程序指定的逻辑。

属性可能是与 schema 相关联的 Git 哈希，一个环境字符串，如dev 或 prod 。

Example

这是字符串的 SchemaInfo。

{
    "name": "test-string-schema",
    "type": "STRING",
    "schema": "",
    "properties": {}
}

Schema 名称

Pulsar 支持各种 schema 模式，这些模式主要分为两类：

  • 原始类型

  • 复杂类型

原始类型

目前，Pulsar 支持以下原始类型：

原始类型Description
BOOLEAN二进制值
INT8一个 8 位符号整数
INT16一个 16 位符号整数
INT32一个 32 位符号整数
INT64一个 64 位符号整数
FLOAT单精度(32位) IEEE 754 浮点数
DOUBLE双精度(64位) IEEE 754 浮点数
BYTES8 位无符号字节的序列
STRINGUnicode 字符序列
TIMESTAMP (DATE, TIME)逻辑类型代表时间内特定的即时精度毫秒。
它储存自 1970年1月1日，00:00:00 GMT 以来的毫秒数为 INT64 值

对于原始类型，Pulsar 不在 SchemaInfo 中存储任何架构数据。 SchemaInfo 中的 类型 用于决定如何序列化和反序列化数据。

一些原始架构实现可以使用 属性 来存储实现特定的调节设置。 例如， 字符串 schema 可以使用 属性 来存储编码集来序列化和反序列化字符串。

The conversions between Pulsar schema types and language-specific primitive types are as below.

Schema 类型Java 类型Python 类型Go 类型
BOOLEANbooleanboolbool
INT8字节int8
INT16shortint16
INT32intint32
INT64longint64
FLOATfloatfloat浮点32
DOUBLE双精度float浮点64
BYTESbyte[], ByteBuffer, ByteBufbytes[]字节
STRINGstringstrstring
TIMESTAMPjava.sql.Timestamp
TIMEjava.sql.Time
DATEjava.util.Date

Example

此示例演示如何使用字符串模式。

  1. 创建一个带有字符串模式的生产者并发送消息。

    Producer producer = client.newProducer(Schema.STRING).create();
producer.newMessage().value("Hello Pulsar!").send();

  2. 使用字符串模式创建消费者并接收消息。

    Consumer consumer = client.newConsumer(Schema.STRING).create();
consumer.receive();

复杂类型

目前，Pulsar支持以下复杂类型：

复杂类型Description
keyvalue表示一个复杂的键/值对。
structSupports AVRO, JSON, and Protobuf.

keyvalue

关键值 schema帮助应用程序定义方案的键值和值。

对于 schemaInfo of keyvalue schema, Pulsar 储存关键架构的 SchemaInfo 和值架构的 SchemaInfo

Pulsar 提供两种方法来编码消息中的密钥/值对：

  • 嵌入

  • 已设置

用户可以在构建密钥/值模式时选择编码类型。

嵌入

键/值对将在消息有效载荷中一起编码。

已设置

密钥将被编码为消息密钥，其值将被编码为消息有效载荷。

Example

这个示例演示如何构建一个键/值模式，然后用它来生成和消费消息。

  1. 构造具有 INLINE 编码类型的密钥/值架构。

    Schema<KeyValue<Integer, String>> kvSchema = Schema.KeyValue(
Schema.INT32,
Schema.STRING,
KeyValueEncodingType.INLINE
);

  2. 可选，构建一个包含 SEPARATED 编码类型的密钥/值方案。

    Schema<KeyValue<Integer, String>> kvSchema = Schema.KeyValue(
Schema.INT32,
Schema.STRING,
KeyValueEncodingType.SEPARATED
);

  3. 使用密钥/值模式生成消息。

    Schema<KeyValue<Integer, String>> kvSchema = Schema.KeyValue(
Schema.INT32,
Schema.STRING,
KeyValueEncodingType.SEPARATED
);

Producer<KeyValue<Integer, String>> producer = client.newProducer(kvSchema)
    .topic(TOPIC)
    .create();

final int key = 100;
final String value = "value-100";

// send the key/value message
producer.newMessage()
.value(new KeyValue<>(key, value))
.send();

  4. 使用密钥/值模式消费消息。

    Schema<KeyValue<Integer, String>> kvSchema = Schema.KeyValue(
Schema.INT32,
Schema.STRING,
KeyValueEncodingType.SEPARATED
);

Consumer<KeyValue<Integer, String>> consumer = client.newConsumer(kvSchema)
    ...
    .topic(TOPIC)
    .subscriptionName(SubscriptionName).subscribe();

// receive key/value pair
Message<KeyValue<Integer, String>> msg = consumer.receive();
KeyValue<Integer, String> kv = msg.getValue();

struct

Pulsar 使用 Avro 规格 声明结构 架构定义 架构。

Pulsar 允许：

  • 使用相同的工具管理 schema 定义

  • 使用不同的序列化/反序列化方法处理数据

有两种方法可以使用 struct schema：

  • 静态

  • 通用

静态

您可以预定义 结构 架构，它可以是 Java 中的 POJO 模式。 a 结构 以Go, 或 Avro 或 Protobuf 工具生成的类。

Example

Pulsar 使用 Avro 库从预定义的 结构 获取架构定义。 schema 定义是作为 SchemaInfo 一部分存储的 schema 数据。

  1. Create the User class to define the messages sent to Pulsar topics.

    public class User {
    String name;
    int age;
}

  2. 创建一个 struct schema 的生产者并发送消息。

    Producer producer = client.newProducer(Schema.AVRO(User.class)).create();
producer.newMessage().value(User.builder().userName("pulsar-user").userId(1L).build()).send();

  3. 创建一个 struct 架构的消费者并接收消息

    Consumer consumer = client.newConsumer(Schema.AVRO(User.class)).create();
User user = consumer.receive();
通用

有时应用程序没有预定义的结构，你也可以使用此方法来定义架构和访问数据。

你可以使用 通用方案生成器来定义 结构 架构。 使用 通用录像生成器 生成一个通用结构，并将消息消耗到 通用记录。

Example

  1. 使用 录制方案生成器 来构建一个方案。

    RecordSchemaBuilder recordSchemaBuilder = SchemaBuilder.record("schemaName");
recordSchemaBuilder.field("intField").type(SchemaType.INT32);
SchemaInfo schemaInfo = recordSchemaBuilder.build(SchemaType.AVRO);

Producer producer = client.newProducer(Schema.generic(schemaInfo)).create();

  2. 使用 录制器 来构建结构记录。

    producer.newMessage().value(schema.newRecordBuilder()
            .set("intField", 32)
            .build()).send();

自动 Schema

如果你事先不知道样式样的 Pulsar 主题类型，你可以使用 AUTO schema 制作或消费通用记录给代理。

Auto Schema 类型Description
AUTO_PRODUCEThis is useful for transferring data from a producer to a Pulsar topic that has a schema.
AUTO_CONSUMEThis is useful for transferring data from a Pulsar topic that has a schema to a consumer.

AUTO_PRODUCE

AUTO_PRODUCE schema 有助于生产者验证生产者发送的字节是否与主题的架构兼容。

Example

假定：

  • You have a producer processing messages from a Kafka topic K.

  • You have a Pulsar topic P, and you do not know its schema type.

  • Your application reads the messages from K and writes the messages to P.

In this case, you can use AUTO_PRODUCE to verify whether the bytes produced by K can be sent to P or not.

Produce pulsarProducer = client.newProducer(Schema.AUTO_PRODUCE())
    …
    .create();

byte[] kafkaMessageBytes = … ; 

pulsarProducer.produce(kafkaMessageBytes);

AUTO_CONSUME

AUTO_CONSUME schema 有助于一个 Pulsar 主题验证由 Pulsar 主题发送的字节是否与消费者兼容， 也就是说，使用 SchemaInfo 从经纪一侧检索到的 Pulsar 主题将消息反序列化为特定语言的对象。

Currently, AUTO_CONSUME only supports AVRO and JSON schemas. 它将消息反序列化为 通用记录。

Example

假定：

  • You have a Pulsar topic P.

  • You have a consumer (for example, MySQL) receiving messages from the topic P.

  • You application reads the messages from P and writes the messages to MySQL.

In this case, you can use AUTO_CONSUME to verify whether the bytes produced by P can be sent to MySQL or not.

Consumer pulsarConsumer = client.newConsumer(Schema.AUTO_CONSUME())
    …
    .subscribe();

Message msg = consumer.receive() ; 
GenericRecord record = msg.getValue();
…

Schema 版本

每个存储主题的 SchemaInfo 有一个版本。 Schema 版本管理一个主题中发生的 schema 更改。

给定的 SchemaInfo 所产生的消息被标记为架构版本， 这样当消息被Pulsar客户端消耗时， Pulsar客户端可以使用schema版本检索相应的 SchemaInfo ，然后使用 SchemaInfo 去反序列化数据。

Schema版本是连续的。 Schema 存储在中间商中，处理相关主题以便进行版本指派。

一旦分配/提供了schema的版本，后续由该生产者生产的所有消息，都将被恰当的版本所标记。

Example

下面的示例说明架构版本是如何工作的。

假设使用 Pulsar Java 客户端 使用下面的代码连接到 Pulsar 并开始发送消息：

PulsarClient client = PulsarClient.builder()
        .serviceUrl("pulsar://localhost:6650")
        .build();

Producer producer = client.newProducer(JSONSchema.of(SensorReading.class))
        .topic("sensor-data")
        .sendTimeout(3, TimeUnit.SECONDS)
        .create();

(3) 任何使用相同模式或主题创建的消费者都可以消耗来自“传感器-数据”主题的消息。

Schema 是如何运行的

Pulsar schemas are applied and enforced at the topic level (schemas cannot be applied at the namespace or tenant level).

生产者和消费者将方案上传给代理，因此 Pulsar schema 在生产者方面和消费者方面工作。

生产者方面

这个图表说明生产者一方的模式工作方式。

  1. 应用程序使用 schema 实例构造生产者实例。

    Schema 实例定义了使用制作者实例生成的数据的 schema 模式。

    以 AVRO 为示例 Pulsar 从 POJO 类中提取 schema 定义，并构建生产者连接时需要传递给代理的 SchemaInfo。

  2. 生产者与代理连接从 SchemaInfo 从已通过的 schema 实例中提取。

  3. 代理在 schema 存储中查找 schema 以检查它是否已经是注册 schema。

  4. 如果是，代理会跳过 schema 验证，因为它是一个已知的 schema 模式，并将 schema 版本退还给生产者。

  5. 如果不是，代理则验证是否可以在此命名空间自动创建模式：

  • If isAllowAutoUpdateSchema sets to true, then a schema can be created, and the broker validates the schema based on the schema compatibility check strategy defined for the topic.

  • If isAllowAutoUpdateSchema sets to false, then a schema can not be created, and the producer is rejected to connect to the broker.

Tip:

isAllowAutoUpdateSchema can be set via Pulsar admin API or REST API.

如何通过 Pulsar admin API设置 isAllowAutoUpdateSchema ，见 管理自动更新战略。

  1. 如果方案被允许更新，则进行兼容策略检查。

  • If the schema is compatible, the broker stores it and returns the schema version to the producer.

    All the messages produced by this producer are tagged with the schema version.

  • If the schema is incompatible, the broker rejects it.

消费者方面

This diagram illustrates how does Schema work on the consumer side.

  1. The application uses a schema instance to construct a consumer instance.

    The schema instance defines the schema that the consumer uses for decoding messages received from a broker.

  2. The consumer connects to the broker with the SchemaInfo extracted from the passed-in schema instance.

  3. 代理决定该话题是否有一个(图案/数据/当地消费者和当地生产者)。

  4. 如果一个主题没有全部内容(图案/数据/当地消费者和当地生产者)：

    • If isAllowAutoUpdateSchema sets to true, then the consumer registers a schema and it is connected to a broker.

    • If isAllowAutoUpdateSchema sets to false, then the consumer is rejected to connect to a broker.

  5. 如果某个话题中有一个主题（架构/数据/本地使用者和本地生产者），则将执行 Schema 兼容性检查。

    • 如果 Schema 通过兼容性检查，那么消费者就将与代理连接。

    • 如果该架构未通过兼容性检查，那么将拒绝使用方连接到代理。

  6. 消费者收到代理的信息。

    如果消费者支持架构的版本控制(例如，AVRO架构)， 消费者获取标签在消息中的版本的 SchemaInfo 并使用已通过的模式和标记在消息中的 schema 来解码消息。
