Schema evolution and compatibility

通常情况下，Schema 在很长时间内不会保持不变。 相反，它们会不断更新变化，以满足新的需求。

本章将探讨 Pulsar Schema 如何演变以及 Pulsar Schema 的兼容性检查战略。

Schema 进化

Pulsar schema is defined in a data structure called SchemaInfo.

Each SchemaInfo stored with a topic has a version. 版本用于管理主题内发生的 schema 更改。

使用 SchemaInfo 生成的消息已标记为架构版本。 当消息被 Pulsar 客户端消耗时， Pulsar 客户端可以使用 schema 的版本检索对应的 SchemaInfo 并使用正确的 schema 信息来反序列化数据。

什么是 Schema 进化？

Schemas 存储属性和类型的详细信息。 为了满足新的业务要求，你需要随着时间的推移不可避免地更新方案，这个方案被称为 schema evolution。

任何 schema 变化都会影响到下游消费者。 Schema 升级确保下游消费者能够无缝地处理以旧方案和新方案编码的数据。

Pulsar schema 应如何演变？

答案是 Pulsar schema 兼容性检查战略。 它决定了schema 如何与主题中的新旧 schema 进行对比。

欲了解更多信息，请参阅 Schema 兼容性检查策略。

Pulsar 是如何支撑 Schema 进化的？

  1. 当生产者/消费者/阅读器连接到代理时，代理会部署由 schemaRegistryCompatibilityCheckers 配置的 schema 兼容性检查器来执行 schema 兼容性检查。

    schema 兼容性检查是每个 schema 类型的一个实例。

    目前，Avro 和 JSON 有自己的兼容性检查器，而所有其他 schema 类型都共享默认兼容性检查器，禁用 schema 进化。

  2. 生产者/消费者/读者将其客户 SchemaInfo 发送给代理。

  3. 代理知道 schema 类型，并且找到此类型的 schema 兼容性检查器。

  4. 代理使用检查器来检查 SchemaInfo 是否与主题的最新 schema 兼容，应用其兼容性检查策略。

    目前，兼容性检查策略是在命名空间级别配置的，适用于命名空间中的所有主题。

Example

  • 示例 1

    在某些情况下，应用程序需要将几种不同类型的事件储存在同一个 Pulsar 主题。

    尤其是，当在 事件来源 风格中开发数据模型时， 你可能有几种影响实体状态的事件。

    例如，对于用户实体来说，有 用户创建, 用户地址变更 和 用户收到了 事件。 应用程序要求这些事件总是按同样顺序阅读。

    因此，这些事件需要进入相同的 Pulsar 分区以维持秩序。 此应用程序可以使用 ALWAYS_COMPATIBLE 来允许不同类型的事件在同一主题中共存。

  • 示例 2

      有时，我们还作出不相容的改变。 
  
  例如，你正在修改一个字段类型从`string`到`int`。
  
  在这种情况下，你需要：

    • 同时将所有生产者和消费者的版本提升到新 schema 版本。

    • 可选，创建一个新主题，并开始迁移应用程序以使用新主题和新模式， 避免需要在同一主题中处理两个不兼容的版本。

BACKWARD 和 BACKWARD_TRANSITIVE

Suppose that you have a topic containing three schemas (V1, V2, and V3), V1 is the oldest and V3 is the latest:

Compatibility check strategyDefinitionDescription

BACKWARD | 使用新 schema 的消费者可以使用 上一版 schema 处理生产者编写的数据。 | 使用 schema V3 的消费者可以处理生产者使用 schema V3 或 V2 编写的数据。 | BACKWARD_TRANSITIVE | 使用新 schema 的消费者可以使用 所有以前的schemas 处理生产者编写的数据。 | 使用 schema V3 的消费者可以处理生产者使用 schema V3、V2 或 V1 编写的数据。 |

Example

  • 示例 1

    删除字段。

    为处理没有一个字段的事件而构建的消费者，可以处理使用包含该字段的旧方案编写的事件。 而消费者会忽略这个领域。

  • Example 2

    你想将所有 Pulsar 数据加载到 Hive 数据仓库中，并对数据运行 SQL 查询。

    即使数据被更改，相同的 SQL 查询也必须继续工作。 为了能够使用它，你可以使用 BACKWARD 策略进化方案。

FORWARD and FORWARD_TRANSITIVE

Suppose that you have a topic containing three schemas (V1, V2, and V3), V1 is the oldest and V3 is the latest:

Compatibility check strategyDefinitionDescription

FORWARD | 消费者使用 前一个 schema 可以处理生产者使用新 schema 编写的数据。 即使他们可能无法充分利用新模式的全部能力。 | 使用schema V3 或 V2 的消费者可以处理生产者使用 schema V3 编写的数据。 | FORWARD_TRANSITIVE | 消费者使用 之前的所有 schemas 可以处理生产者使用新 scheme 编写的数据。 | 使用schema V3、V2 或 V1 的消费者可以处理生产者使用 schema V3 编写的数据。

Example

  • Example 1

    添加字段。

    在大多数数据格式中，即使消费者收到含有新字段的新事件，他们也可以继续处理没有新字段的事件。

  • Example 2

    如果一个消费者有一个应用程序逻辑与一个完整版本的架构， 当 schema升级时，应用程序逻辑可能无法立即更新。

    在这种情况下，你需要将一个新 schema 的数据投射到应用程序理解的旧 schema 上。

    因此， 你可以使用 FORWARD 策略来进化 schemas 以确保旧 schema 能够处理使用新schema 编码的数据。

Schema 验证

当生产商或消费者试图连接到某个主题时，代理会进行一些检查来验证一个方案。

Producer

当生产者试图连接到某个主题时(假定忽略模式自动创建)，代理会进行以下检查：

  • 检查生产者携带的 schema 是否存在于 schema 注册表中。

    • 如果 schema 已经被注册，那么生产商就与经纪人连接并生成与此 schema 方案相关的信息。

    • 如果 schema 没有注册，那么 Pulsar 会验证是否允许根据配置的兼容性检查策略对 schema 进行注册。

Consumer

当消费者试图连接到某个主题时， 代理会根据配置的 schema 兼容性检查策略检查所携带的 schema 是否与注册 schema 兼容。

Order of upgrading clients

The order of upgrading client applications is determined by the compatibility check strategy.

For example, the producers using schemas to write data to Pulsar and the consumers using schemas to read data from Pulsar.

Compatibility check strategy

Upgrade first

Description

ALWAYS_COMPATIBLE

Any order

The compatibility check is disabled.

Consequently, you can upgrade the producers and consumers in any order.

ALWAYS_INCOMPATIBLE

None

The schema evolution is disabled.

  • BACKWARD

  • BACKWARD_TRANSITIVE

Consumers

There is no guarantee that consumers using the old schema can read data produced using the new schema.

Consequently, upgrade all consumers first, and then start producing new data.

  • FORWARD

  • FORWARD_TRANSITIVE

Producers

There is no guarantee that consumers using the new schema can read data produced using the old schema.

Consequently, upgrade all producers first to use the new schema and ensure that the data already produced using the old schemas are not available to consumers, and then upgrade the consumers.

  • FULL

  • FULL_TRANSITIVE

Any order

There is no guarantee that consumers using the old schema can read data produced using the new schema and consumers using the new schema can read data produced using the old schema.

Consequently, you can upgrade the producers and consumers in any order.
